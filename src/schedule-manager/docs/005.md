I think you're starting to discover that you're not building a timer anymore.

You're designing something closer to:

```text
Personal Scheduling Platform
```

and that's good because it changes how we should think about the configuration.

---

# First observation

I would separate three concepts:

```text
Schedule
Event
Notification
```

Right now your event contains notification settings:

```yaml
title: Calculus
start: 08:00
end: 09:30
notification_before_minutes: 10
notification_method: telegram
```

but these are actually different concerns.

---

# Schedule

Schedule answers:

```text
WHEN does something happen?
```

Example:

```yaml
schedule:
  name: University
  cycle_weeks: 2
  start_date: 2026-09-01
```

---

# Event

Event answers:

```text
WHAT happens?
```

Example:

```yaml
event:
  title: Calculus
  start: 08:00
  end: 09:30
```

---

# Notification

Notification answers:

```text
HOW should I be informed?
```

Example:

```yaml
notification:
  trigger_before_minutes: 10
  methods:
    - sound
    - telegram
```

---

This separation becomes very important later.

---

# Timezone

I agree with your idea.

I would make timezone optional.

```yaml
schedule:
  timezone: America/New_York
```

If missing:

```text
Use system timezone
```

This feels natural.

---

# Notification Methods

I would avoid:

```yaml
notification_method: popup|email|telegram|sound
```

because this is a string pretending to be a list.

Instead think:

```yaml
methods:
  - sound
  - telegram
```

Later:

```yaml
methods:
  - sound
  - telegram
  - email
  - popup
```

No schema changes.

---

# Notification Profiles

I think you'll want reusable notifications.

Example:

```yaml
notifications:
  default:
    before_minutes: 10
    methods:
      - sound

  important:
    before_minutes: 30
    methods:
      - sound
      - telegram
```

Then events reference them.

```yaml
title: Exam
notification_profile: important
```

instead of repeating notification settings everywhere.

---

# Event Library

I suspect you'll eventually want this too.

Instead of:

```yaml
week_a:
  monday:
    - title: Calculus
```

you define:

```yaml
events:
  calculus:
    title: Calculus

  physics:
    title: Physics
```

and schedule references them.

```yaml
monday:
  - event: calculus
    start: 08:00
    end: 09:30
```

This allows:

- icons
- colors
- descriptions
- URLs
- locations

to live in one place.

---

# Metadata

I would plan for this from day one.

Example:

```yaml
events:
  calculus:
    title: Calculus

    metadata:
      professor: Smith
      room: B201
      building: Science Hall
```

Then the TUI can show:

```text
Current Event

Calculus
Room B201
Professor Smith
```

---

# Colors

Textual will eventually want colors.

I would reserve:

```yaml
color: blue
```

or

```yaml
color: "#3B82F6"
```

for events.

Example:

```yaml
events:
  physics:
    title: Physics
    color: blue

  math:
    title: Mathematics
    color: green
```

Then the dashboard becomes visually useful.

---

# Telegram Integration

I would not put Telegram details inside schedules.

Bad:

```yaml
telegram_chat_id: ...
```

inside every schedule.

Instead:

```yaml
config/

app.yaml
notifications.yaml

schedules/
university.yaml
work.yaml
```

Application settings:

```yaml
telegram:
  enabled: true
  bot_token: ...
  chat_id: ...
```

Schedules remain portable.

# What I think V1 should support

Only these concepts:

```text
Schedule
├── id
├── name
├── start_date
├── cycle_weeks
├── timezone (optional)
└── weeks

Week
└── weekdays

Day
└── event instances

Event Instance
├── title
├── start
├── end
├── color (optional)
└── notification profile

Notification Profile
├── before_minutes
└── methods
```

Nothing more.

No FastAPI.
No Web UI.
No Telegram Bot commands.
No database.

Just:

```text
YAML
+
Schedule Engine
+
Textual Dashboard
+
Notification Engine
```

That's already a surprisingly powerful system and a very reasonable first milestone. The FastAPI server and Telegram integration can sit on top of exactly the same scheduling engine later.
