# voice-generator Orpheus Config Completion

## Summary

Finished the Orpheus configuration path so the backend can be driven from
config files or environment variables instead of only CLI flags.

## Files Changed

- `src/voice-generator/README.md`
- `src/voice-generator/docs/DEVELOPMENT_GUIDELINE.md`
- `src/voice-generator/docs/IMPLEMENTATION_PLAN.md`
- `src/voice-generator/docs/reports/report_2026-06-30_06-02-49.md`
- `src/voice-generator/tests/test_config.py`
- `src/voice-generator/tests/test_registry.py`
- `src/voice-generator/voice_generator/cli.py`
- `src/voice-generator/voice_generator/config.py`

## Behavior Changed

- Orpheus runtime command, model path, voice catalog, and command template can
  be set from config files.
- Environment variables now override config defaults for Orpheus and other
  core path settings.
- `voice providers` now reflects the loaded config, so Orpheus shows `ready`
  when its runtime command and model path are configured.
- The config layer now validates blank override values before the CLI reaches
  provider dispatch.

## Checks Run

- `uv run python -m compileall src/voice-generator/voice_generator src/voice-generator/tests`
- `uv run python -m unittest discover -s src/voice-generator/tests`
- `uv run voice-generator providers`
- `uv run python -c "<config-backed Orpheus providers smoke test>"`

## Remaining Risks

- Orpheus still depends on a user-supplied runtime command and GGUF model.
- The runtime command template is intentionally generic, so upstream prompt and
  decoding details are still handled by the configured backend.
- Kokoro, Piper, and ElevenLabs remain unimplemented.
